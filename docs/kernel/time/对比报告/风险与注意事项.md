## 第五部分：风险与注意事项

### 5.1 技术风险

#### **1. TSC 可靠性**

**问题**：
- 老 CPU 的 TSC 可能随频率变化
- C-state 暂停 TSC
- SMP 多核 TSC 不同步

**解决方案**：
- 检测 `Invariant TSC` (CPUID.80000007H:EDX[8])
- 使用 watchdog 机制验证
- 降级到 HPET/ACPI PM

#### **2. HPET 可用性**

**问题**：
- 某些虚拟化环境 HPET 不稳定
- HPET 寄存器访问延迟较高
- 部分 BIOS 禁用 HPET

**解决方案**：
- 保留 ACPI PM 作为最终备选
- 检测 HPET 是否可用
- 动态调整 rating

#### **3. 中断延迟**

**问题**：
- 高精度定时器需要频繁中断
- 中断延迟影响精度

**解决方案**：
- 批量处理 hrtimer 到期
- 使用 tickless 减少 wakeups
- 优化中断处理路径

### 5.2 兼容性风险

#### **1. 用户空间 ABI**

**现有 POSIX API**：
- `clock_gettime`
- `timer_create`
- `nanosleep`

**风险**：
- 高精度模式可能导致行为差异
- 某些程序依赖旧精度

**解决方案**：
- 保持 API 兼容
- 内部透明切换高精度模式
- 提供 `/proc/sys` 开关

#### **2. 驱动程序兼容性**

**问题**：
- 某些驱动直接访问时钟寄存器
- 可能依赖旧的时钟源

**解决方案**：
- 统一通过 timekeeping API
- 标记旧接口为 `deprecated`

### 5.3 性能风险

#### **1. 锁竞争**

**问题**：
- 时钟读取是高频操作
- 当前实现有锁竞争

**解决方案**：
- 实现 seqcount 快速路径
- per-CPU 变量优化
- 减少临界区

#### **2. 缓存友好性**

**问题**：
- `ClocksourceData` 使用 `String` 类型
- 动态分配导致缓存失效

**解决方案**：
- 使用 `&'static str` 替代 `String`
- 优化数据结构布局
- 使用 `#[repr(C)]` 保证对齐
